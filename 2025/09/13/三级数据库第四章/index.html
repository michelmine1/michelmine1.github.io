三级数据库第四章——数据库应用系统功能设计与实施

一、数据完整性保护

数据完整性包括实体完整性、参照完整性和用户自定义完整性。

①实体完整性在 CREATE TABLE 中用 PRIMARY KEY （主键）定义；

②参照完整性在 CREATE TABLE 中用 FOREIGN KEY 短语定义哪些列为外码，用 REFERENCES 短语指明这些外码参照哪些表的主码；

③用户自定义完整性在 CREATE TABLE 中定义属性的同时也可以根据应用要求，定义属性上的约束条件，即属性值限制，包括：列值非空（NOT NULL），列值唯一（UNIQUE），检查列值是否满足一个布尔表达式（CHECK 短语）。

二、应用服务器

应用服务器是指通过各种协议把商业逻辑提供给客户端的程序。

它提供了访问商业逻辑的途径以供客户端应用程序使用，并接收来自于 Web 浏览器的用户请求，根据应用领域业务规则执行相应的数据库应用程序，通过访问接口向数据库服务器提出数据操作请求；它亦接收来自于数据库服务器的数据访问结果，并通过客户端将结果返回用户。所以当数据库服务器出现问题时无法向用户提供服务。

三、事务

事务是并发控制的基本单位。

所谓的事务，它是一个操作序列，这些操作要么都执行，要么都不执行，它是一个不可分割的工作单位。

在数据库应用系统设计中，事务的特性之一就是一致性，即在和用户的交互过程中，保证数据的一致性。其他三项都是事务长时间和用户交互带来的坏处。当事务长时间和某个用户进行交互时，导致其他用户需要等待该事务结束以获取资源，因此会在一定程度上降低了不同事务的并发度，同时因为长时间的交互导致资源被用户占用的时间过长，将会增大死锁的可能性，系统在执行检查点操作时，所有的事务处理都被暂时中止，长时间的事务会导致检查点的效果。

四、从功能角度划分的数据库应用系统

数据应用系统从功能的角度可以划分为四层，分别是表示层、业务逻辑层、数据访问层和数据持久层。

①表示层：用于显示和接收用户输入的人机（一般为 Windows 应用程序或 Web 应用程序），即人机界面设计。

②业务逻辑层：是表示层和数据访问层之间的桥梁，主要负责数据的传递和处理。梳理DBMS的各项业务活动，将其表示为各种系统架构。

③数据访问层：实现对数据的保存和读取操作。可以访问关系数据库、文本文件或 XML 文档等。

④数据持久层：以程序为媒介把表现层或业务层的数据持久化，它位于物理层和数据访问层之间。

五、三层B/S（浏览器/服务器）结构

B/S结构的数据库应用系统得到特点是用户界面完全用WWW浏览器来实现，一部分事务逻辑在前端实现，主要的事务逻辑在服务器实现，所以对其客户机配置要求不高，即使对服务器要求高，也不需要安装客户端软件。数据处理功能分别放在表示层、功能层和数据层三个层次上。表示层位于客户端，由 Web 浏览器组成；功能层位于 Web 应用服务器，实现面向具体应用领域的业务规则；数据层位于数据库服务器，按照事务逻辑对数据库进行操作。在 B/S 结构中，业务逻辑在应用服务器，事务逻辑在数据库服务器。

三层架构将更多的任务传给服务器端计算，所以增加了网络的通信量，其运行速度受制于网络，并不一定会提高。在适用Internet、维护工作等等方面，B/S比C/S强；在运行速度、数据安全、人机交互等方面，不如C/S（客户端/服务器）。

六、三层C/S（客户端/服务器）架构

C/S 结构是客户机 / 服务器结构，客户端负责完成用户的交互功能，接收用户数据，根据业务规则处理应用任务，向数据库服务器发送请求，数据库服务器则通过 DBMS，依据事务逻辑完成相应的数据管理功能。因此，在 C/S 结构中，业务逻辑在客户端。事务逻辑在数据库服务器。

七、封锁

封锁是指在某一时间内禁止某用户对数据对象做某些操作以避免产生数据不一致的问题。封锁粒度与系统的并发度和并发控制的开销密切相关。封锁的粒度越大，并发度就越小，同时系统的开销就越小。封锁的粒度越小，并发度就越大，同时系统的开销就越大，所以提高封锁的粒度带来的好处是减少了封锁开销。

八、事务概要设计

事物概要设计内容包括：事务名称、事务所访问的关系表及关系属性、事务处理逻辑、事务用户（指使用、启动、调用改事务额软件模块或系统）。

九、约束的完整性级别

  1. 列级完整性约束

    定义：针对表中单个列（字段）上的数据设置的约束条件。它仅影响这一列。

    常见类型：

    • NOT NULL：约束该列的值不能为空。
    • UNIQUE：约束该列的值必须是唯一的（但允许多个NULL值，除非同时被NOT NULL约束）。
    • DEFAULT：为该列设置默认值。当插入数据未指定该列值时，自动使用默认值。
    • CHECK：检查该列的值是否满足一个指定的条件（布尔表达式）。
    • PRIMARY KEY：主键约束，是 NOT NULL 和 UNIQUE 的结合。虽然主键定义在表级，但其约束力作用于单个列（对于单列主键）。

  2. 元组级完整性约束

    定义：针对表中一行数据（一个元组） 中多个列之间的逻辑关系设置的约束条件。它用于约束同一行内不同列的值必须满足某种条件。

    常见实现：

    • CHECK 约束：在表定义时使用CHECK子句，其中的条件可以涉及多个列。

  3. 数据库级完整性约束

    定义：这是一个更宽泛的概念，指任何需要从整个数据库的视角来考虑的复杂约束。它可能涉及多个表、复杂的CHECK条件或业务规则。

    常见实现：

    1. 复杂的CHECK约束：虽然CHECK可用于元组级，但某些数据库系统（如PostgreSQL）支持跨表的CHECK约束（通过函数实现），但这并非SQL标准。
    2. 触发器：这是实现数据库级完整性约束最主要和最灵活的工具。当简单的CHECK或FOREIGN KEY无法表达复杂规则时，就使用触发器。
    3. 存储过程：通过强制所有数据修改都通过特定的存储过程进行，在过程中实现复杂的业务逻辑校验。

  4. 关系级完整性约束

    定义：主要用于维护表与表之间的关系，最常见的就是外键约束。

    常见类型：

    • FOREIGN KEY：确保一个表（子表）中的某个列的值必须存在于另一个表（父表）的指定列中。它定义了表间的引用关系。

十、死锁

1、定义

死锁是指两个或两个以上的进程（或线程，在数据库中也叫事务）在执行过程中，由于资源竞争或彼此通信而造成的一种互相等待的现象。若无外力干涉，这些进程都无法向前推进。

2、一个经典的比喻：哲学家就餐问题

这是解释死锁最著名的思想实验：

  1. 有五位哲学家围坐在一张圆桌旁，桌上有五根筷子（每两位哲学家之间放一根）。
  2. 哲学家的生活只有两种活动：思考和吃饭。
  3. 吃饭时需要同时拿起左手边和右手边的两根筷子。
  4. 如果一位哲学家拿起了左手边的筷子，但右手边的筷子正被旁边的人拿着，那么他就必须等待，直到拿到另一根筷子才能开始吃饭。

现在，想象一下这个场景：

所有哲学家同时感到饥饿，他们每个人都同时拿起了自己左手边的筷子。

此时，每根筷子都被占用了。每个哲学家都在等待自己右手边的筷子被释放，但没有一个哲学家能够拿到右手边的筷子，因为所有筷子都被拿走了。

结果就是：所有哲学家都拿着一根筷子，永远在等待另一根，谁也吃不上饭。这就是死锁。

3、死锁产生的四个必要条件（Coffman Conditions）

这四个条件必须同时满足，死锁才会发生。只要打破其中任意一个，就能预防死锁。

  1. 互斥：一个资源每次只能被一个进程使用。
    • （筷子一次只能被一位哲学家使用。）
  2. 请求与保持：一个进程因请求资源而阻塞时，对已获得的资源保持不放。
    • （哲学家拿着左手的筷子，同时在等待右手的筷子，不会放下左手的。）
  3. 不可剥夺：进程已获得的资源，在未使用完之前，不能被其他进程强行剥夺。
    • （你不能从哲学家手里强行把筷子抢走。）
  4. 循环等待：若干进程之间形成一种头尾相接的循环等待资源关系。
    • （哲学家A在等哲学家B的筷子，哲学家B在等哲学家C的筷子，……，哲学家E又在等哲学家A的筷子，形成了一个循环。）

4、如何解决死锁？

解决死锁的策略主要分为三种：

  1. 死锁预防
    • 思路：破坏死锁产生的四个必要条件中的一个或多个。
    • 方法：
      • 各个事务应顺序访问共享资源，将复杂的、大的事务分解成多个简单、小的事务
      • 按同一顺序访问对象
      • 事务隔离性级别
      • 使用绑定连接
      • 灵活使用显示加锁技术
  2. 死锁检测与恢复
    • 思路：允许死锁发生，但系统会定期检测是否存在死锁。一旦检测到，就采取行动解除它。
    • 方法：
      • 检测：数据库管理系统会维护一个“等待图”。如果在这个图中发现了一个环，就表明发生了死锁。
      • 恢复：选择一个“牺牲品”事务，将其回滚，并释放它持有的所有锁。这样其他事务就可以继续进行了。这个选择策略可能是回滚代价最小的事务，或已经执行时间最短的事务。
  3. 死锁忽略
    • 思路：假设死锁很少发生，直接忽略它。如果真发生了，就等待超时。
    • 方法：为每个请求设置一个超时时间。如果一个请求等待锁的时间过长，就简单地认为它可能遇到了死锁，然后将其回滚。这种方法在现代数据库中使用较少，因为超时时间难以设定，且效率较低。

在现代关系型数据库（如MySQL, PostgreSQL, Oracle, SQL Server）中，通常内置了强大的死锁检测机制。 一旦它们检测到死锁，会立即选择一个事务进行回滚，并向应用程序返回一个死锁错误。应用程序收到这个错误后，可以重试被回滚的事务。

十一、业务逻辑层概要设计的原则

  1. 构件本身应由相关性很强的代码组成，一个构件或一个模块只负责完成一项任务，也就是常说的单一责任原则。
  2. 组成系统业务逻辑层的各个构件应具有独立的功能，并且最大限度地减少与其他构件功能重叠
  3. 构件之间的接口应尽量简单明确
  4. 如果某两个构件间的关系比较复杂，应考虑进一步进行模块划分
  5. 如果构件过于复杂，可以将其细分。

十二、数据存储安全

数据是所有信息系统的基础，如何保障数据安全也是信息系统设计和建设过程中需要考虑的首要问题。数据安全涉及的范围很广，从数据存储安全角度出发主要有以下五个方面：

①安全性保护：即防止非法用户对数据库的非法使用，以避免数据的泄露、篡改或破坏。

②完整性保护：即保证数据的正确性和一致性。

③并发控制：即保证多个用户能共享数据库，并维护数据的一致性

④数据库的备份与恢复：即在系统失效后数据库恢复，配合定时备份数据库，使数据库不丢失数据。

⑤数据加密传输：即将一些高密级的敏感数据通过一定的加密算法加密后传输。
环境安全设计包括漏洞和补丁、计算机病毒防护、网络环境安全和物理环境安全。

十三、关系表和元组

1、关系表在逻辑上由一系列元组组成，元组由多个属性组成

2、每个元组可以用数据库磁盘文件中的一个逻辑记录来存储，记录包括多个域

3、元组的每个属性对应于文件记录中的一个域

十四、两段锁协议

两段锁协议是指所有事务必须分两个阶段对数据项加锁和解锁：

  1. 在对任何数据进行读、写操作之前，要申请并获得对该数据的封锁
  2. 在释放一个封锁之后，事务不再申请和获得其它任何封锁

可以证明，若并发执行的所有事务均遵守两段锁协议，则对这些事务的任何并发调度策略都是可串行化的。

另外要注意两段锁协议和一次封锁法的异同之处。一次封锁法要求每个事务必须一次将所有要使用的数据全部加锁，否则就不能继续执行，因此一次封锁法遵守两段锁协议。但两段锁协议并不要求事务必须一次将所有要使用的数据全部加锁，因此遵守两段锁协议的事务可能发生死锁。加锁阶段事务可以申请获得任何数据项上的任何类型的锁，但是不允许释放任何锁。

十五、事务的四大特性（ACID）

  1. 原子性（Atomicity）：事务中的操作“要么全做，要么全部不做”，是一个不可分割的整体。
  2. 一致性（Consistency）：事务执行前后，数据库从一个“一致状态”转移到另一个“一致状态”。
  3. 隔离性（Isolation）：多个十五并发执行时，一个事务的执行不能被其他事务干扰。
  4. 持久性（Durability）：事务提交后，对数据库的修改是“永久的”，即使系统崩溃也不会丢失。

三级数据库技术1.数据库的三级模式结构和二层映像功能提供了数据的逻辑独立性和物理独立性，其中外模式/模式映像提供逻辑独立性，模式/内模式提供物理独立性。 内模式（最低层）：反映了数据在计算机物理结构中的实际存储形式。内模式也称存储模式或物理模式，它是数据物理结构和存储结构的描述，是数据在数据库内部的表示方式。并且一个数据库只有一个内模式。 模式（中层）：反映了设计者的数据全局逻辑要求。模式也称逻辑模式，是数据库中全体数据的逻辑结构和特征的描述，是所有用户的公共数据视图。它是数据库系统模式结构的中间层，不涉及数据的物理存储细节和硬件环境，与具体的应用和所使用的开发工具及高级程序设计语言无关。 外模式（最外层）：反映了用户对数据的要求。外模式也称子模式或用户模式，它是数据库用户（包括应用程序员和最终用户）看见和使用的局部数据的逻辑结构和特征的描述，是数据库用户的视图，是与某一应用有关的数据的逻辑表示。外模式是模式的子集，一个数据库可以有多个外模式。 2.概念数据模型：是数据库设计人员在设计的初始阶段摆脱计算机系统和DBMS的具体技术问题，集中精力分析数据以及...
