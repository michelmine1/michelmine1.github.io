三级数据库第三章——数据库结构设计

一、.对模式进行分解

对模式进行分解时，既要保证分解具有“无损连接性”，又要保证分解“保持函数依赖”。所谓“无损连接”指分解的若干连接重组时可以精确恢复到原来的数据表，数据记录既没有增加也没有减少。“保持函数依赖”是指原关系模式含有的属性之间的隐含关系在分解后不能丢失。

1.关系(relation)：就是日常所说的“表”。

​ 一个“关系”由两部分组成：

  • 关系模式：表的结构或蓝图。定义了表的名字、列（属性）、列的数据类型以及约束条件。

    格式：关系名（属性1，属性2，…，属性n）

    例子：学生（学号、姓名、系别）

  • 关系实例：表的内容，即存在表内的数据，是一系列“元组”（行）的集合。

2.依赖（dependency）:关系中属性之间的某种约束关系。最重要的类型是函数依赖。

  • 函数依赖：如果在一个关系中，对于属性集X的每一个值，都唯一地确定属性集Y的值，那么就称Y函数依赖于X，或者说X函数决定Y，记作X->Y。（X称为决定因子，Y称为被决定因子）

    理解：就像函数y=f(x)一样，给定一个x，只能得到唯一的一个y。

    例子：学生表中，一个学号唯一对应一个学生的姓名。这是一种强约束。

    ​ 一个学号唯一对应一个学生所在的系。

    ​ 一个姓名不能唯一确定一个系，所以关系不成立。

3.分解（decomposition）：是将一个包含多个属性的关系模式（一张大表）拆分成多个包含更少属性的关系模式（一张小表）的过程。

  • 目的：

    消除数据冗余：相同的数据不需要在多个地方重复存储。

    避免操作异常：

    • 插入异常：想插入一部分信息，但因为主键为空而无法插入。
    • 删除异常：删除一条信息时，不小心把其他不想删除的信息也删掉了。
    • 更新异常：更新一个地方的数据，但相同数据存储在多处，导致数据不一致。

​ 分解必须遵循的原则：
​ 分解不是简单地把列随便分到不同的表里。一个至关重要的原则是“无损连接分解”，意思是分解后 的多个小表通过自然连接操作，能够精确地、不丢失也不增加任何信息地重建原来的大表。

二、数据模型的三要素

  1. 数据结构

    数据结构是所研究的对象类型的集合。它从语法角度表述了客观世界中数据本身的结构和数据对象之间的关联关系，是对系统静态特征的描述。

  2. 数据操作

    数据操作是对数据库中对象的实例允许执行的操作的集合，主要指检索和更新（插入、删除、修改）两类操作。数据模型必须定义这些操作的确切含义、操作符号、操作规则（如优先级）以及视线操作的语言。数据操作是对系统动态特征的描述。

  3. 数据完整性约束

    数据完整性约束是一组完整性规则的集合，规定数据库状态及状态变化所应满足的条件，以保证数据的正确性、有效性和相容性。

三、E-R图向关系模式转换

ER 图向关系模式转换涉及到两方面：实体的转换和实体间联系的转换。

①实体的转换：在从 ER 图转换为关系模式时，一个实体就转换成一个关系模式，实体的属性就是关系模式的属性，实体的键就是关系模式的主键。
②实体间联系的转换：实体间存在三种联系即 1:1（一对一）联系、1:m（一对多）联系，m:n（多对多）联系。在从 ER 向关系模式转换规则如下：
1:1（一对一）联系
方法一：联系转换为独立的关系模式；模式的属性由联系本身的属性及两个实体的键构成；主键由两个实体中的任意一个键构成。
方法二：联系与一端的实体的关系模式合并，即将联系的属性加入到实体的关系模式内，主键不变。
1:m（一对多）联系
方法一：联系转换为独立的关系模式；模式的属性由联系本身的属性及两个实体的键构成；主键由 m 端实体的键组成。
方法二：与 m 端的实体的关系模式合并，即将联系的属性加入到实体的关系模式内，主键不变。
m:n（多对多）联系
多对多联系转换成新的独立的模式时，模式的属性由联系本身的属性及两个实体的键构成，主键由两端实体的键组合而成。

在 ER 图中，将实体和联系转换成关系模式的规则是：
（1）每一个实体集转换为一个关系模式；
（2）一对一、一对多联系中一端并入多端的实体关系模式中；
（3）每一个多对多联系转换成一个关系模式。

具体转换方法可看链接：【数据库必考】怎么画ER图及如何转为关系模型_哔哩哔哩_bilibili

结论：ER图转向关系模型，关系模型的个数=实体数+多对多关系

四、一张表的主码

在关系表中，主键需非空且唯一，外码可取被参照关系已有的主键值或空值。

五、IDEF1X数据建模方法

1、IDEF1X 是 IDEF 系列方法中 IDEF1 的扩展版本，是在 E - R（实体联系）方法的原则基础上，增加了一些规则，使语义更为丰富的一种方法。概念模型设计常用 IDEF1X 方法。IDEF1X 建模方法：实体集（独立实体集、从属实体集）、联系（标定型联系、非标定型联系、分类联系、非确定联系）。从属实体集可以有自己的主码，并不是只能参与标定型联系。

2、四种联系方式：

  • 标定联系
    • 定义：在这种联系中，子女实体的每个实例都由它与双亲实体的联系而确定，子女实体的主键部分或全部由双亲实体的主键构成 。可以理解为，没有双亲实体，子女实体就无法完整定义 。
    • 举例：比如在 “部门 - 员工” 关系中，假设一个员工只能属于一个部门。员工是子女实体，部门是双亲实体，员工表中的主键可能包含部门编号（部门表的主键）以及员工自己的编号，员工的唯一性需要结合所属部门来确定。
  • 非标定联系
    • 定义：子女实体有自己独立的主键，它与双亲实体的联系不影响其主键的构成 。双亲实体与子女实体之间是一种 “拥有” 的关系，子女实体可以独立存在，即使没有双亲实体的参照，也能唯一标识自身实例 。
    • 举例：以 “公司 - 车辆” 关系为例，公司是双亲实体，车辆是子女实体。每辆车有自己独立的车牌号作为主键，即便不考虑所属公司，通过车牌号也能唯一确定一辆车 。
  • 分类联系
    • 定义：一个实体集（称作分类实体集）是另一个实体集（称作被分类实体集）的子集合。被分类实体集的每个实例恰好是分类实体集中一个且仅一个实例 ，本质上是一种 “父类 - 子类” 的关系 。
    • 举例：在 “动物 - 哺乳动物、鸟类” 关系中，动物是被分类实体集，哺乳动物和鸟类是分类实体集。哺乳动物和鸟类都是动物这个大集合下的子类，每一种具体的动物（如老虎属于哺乳动物，麻雀属于鸟类）都能明确划分到某一个分类实体集中 。
  • 非确定联系
    • 定义：又称为 “多对多” 联系，指两个实体集之间，任一方的一个实例都能与另一方的多个实例相关联 。在实际的数据库设计中，通常需要引入一个中间实体来将非确定联系转化为两个一对多的联系 。
    • 举例：“学生 - 课程” 关系，一个学生可以选修多门课程，一门课程也可以被多个学生选修。为了在数据库中表示这种关系，通常会创建一个 “选课” 中间表，记录学生与课程之间的选课对应关系 。

五、索引

一个数据文件上可以建立多个索引文件。如果数据文件中数据记录的排列顺序与索引文件中索引项的排列顺序一致，则称为该索引文件为聚集索引，否则称为非聚集索引。一个数据文件上只能建立一个聚集索引，可以建立多个非聚集索引。

如果频繁地对数据进行增加、删除和更改操作，则系统会花费很多时间来维护索引，这会降低数据的插入和修改效率。另外，存储索引需要占用额外的空间，这增加了数据库的空间开销。因此，要根据表和数据量的情况，确定如何构建索引。当不需要某个索引时，可将其删除。

  • 在数据文件的主码属性集上建立的索引称为主索引。主索引可以是聚集索引，也可以是非聚集索引。
  • 对数据文件和它的一个特定的索引文件，如果数据文件中数据记录的排列顺序与索引文件中索引项的排列顺序相一致，则称该索引文件为聚集索引。聚集索引可以是主索引，也可以是辅索引。
  • 有序索引技术利用索引文件实现记录域取值到记录物理地址间的映射关系。有序索引适用于查询某个范围的情况，也适用于点查询。
  • 散列技术利用一个散列函数实现记录域取值到记录物理地址间的直接映射关系。散列索引适用于查询某一个特定值的情况。

七、数据库的物理设计阶段

1、物理设计涉及底层存储、存取，与操作系统、硬件、DBMS 密切相关

2、物理设计要考虑 DBMS 和操作系统的相关参数。

3、物理设计需考虑 RAID 级别（影响存储可靠性和性能）、操作系统文件管理机制、DBMS 支持的索引类型等。

4、需要合理安排不同的存储介质，日志文件对于数据库的恢复等操作非常关键，需要快速的读写速度，通常应存储在高速磁盘中，而不是磁带（磁带读写速度慢，不适合日志文件这种需要及时记录和访问的情况）

八、第一范式、候选码、依赖等

如果关系模式 R 中不包含多值属性，则 R 满足第一范式，简称 1NF，记作 R ∈ 1NF。易知此题中 R ∈ 1NF，且 R 的所有非主属性不依赖于一个候选码，候选码（A、C）→（B、D）存在非主属性对主码部分函数依赖，故不满足 2NF，所以 R 最高 1NF。

举例：

学号 姓名 系别 系主任 课程编号 课程名称 成绩
101 张三 计算机系 李主任 （C01,C02） （数据库，数据结构） （95,88）
102 李四 物理系 王主任 （C03） （电磁学） （92）

1、第一范式

定义：第一范式要求数据库表中的所有字段都是不可再分的原子值。

这意味着表中的每一列都只能包含单一的值，不能是集合、数组或记录等非原子结构。换句话说，一个单元格里只能有一个值。

所以学生课程表违反了第一范式，上述表格的“课程编号”“课程名称”“成绩”列包含了多个值（集合）

2、依赖

依赖主要描述的是属性（列）之间的关系。最重要的两种依赖是函数依赖和完全函数依赖。

a.函数依赖

定义：在一个关系表R中，如果知道一个属性集A的值，就可以唯一确定另一个属性集B的值，则称B函数依赖于A，记作 A → B。

解释：

  • A 称为决定因子。
  • 可以通俗地理解为：A 就像主键一样，B 的值是由 A 决定的。
  • 注意：函数依赖是基于数据本身的语义（业务逻辑）来确定的，而不是看当前表里有没有重复值。它是对现实世界的一种约束。

在我们的例子中：

  • 学号 → 姓名：因为一个学号只对应一个学生姓名。
  • 学号 → 系别：因为一个学生只属于一个系。
  • 系别 → 系主任：因为一个系只有一个系主任（在这个业务设定下）。
  • 课程编号 → 课程名称：因为一门课程有一个固定的名称。
  • (学号, 课程编号) → 成绩：因为一个学生的一门课程只有一个成绩。

b. 完全函数依赖

定义：在函数依赖 A → B 中，如果B依赖于A的整个集合，而不是A的任何一个真子集，则称B完全函数依赖于A。

解释：
这是为了区分部分依赖（Partial Dependency）。部分依赖是指B依赖于A的某一个子集。

在我们的例子中：

  • (学号, 课程编号) → 成绩 是一个完全函数依赖。因为“成绩”既不能单独由“学号”决定，也不能单独由“课程编号”决定，必须由它们两者共同决定。
  • (学号, 课程编号) → 姓名 是一个部分依赖。因为其实只需要“学号”就能决定“姓名”，“课程编号”在这里是多余的。即：学号 → 姓名。

部分依赖是导致数据冗余的关键原因之一，也是第二范式要消除的对象。

3、候选码

定义：候选码是能够唯一标识表中每一行的最小属性集合。

解释：

  • 唯一性：候选码的值必须是唯一的，不能重复。
  • 最小性：候选码的属性集合中，不能移除任何一个属性，否则将破坏唯一性。也就是说，候选码中不应该包含无关紧要的属性。
  • 一个表可以有多个候选码。

在我们的例子中：

  • 属性集 (学号, 课程编号) 可以唯一确定一行（一个学生的一门课程成绩）。如果只使用 学号，那张三的两行记录就无法区分；如果只使用 课程编号，C01课程被多个学生选修时也无法区分。所以 (学号, 课程编号) 是一个候选码。
  • 假设“姓名”也是唯一的（没有重名学生），那么 (姓名, 课程编号) 也可以作为一个候选码？但通常姓名不能保证绝对唯一，所以在这个例子中，我们通常认为只有一个候选码：(学号, 课程编号)。

相关概念：主键

  • 主键：从多个候选码中，由数据库设计者选定一个用来作为表的唯一标识符，这个被选中的候选码就称为主键。
  • 主键就是“正在服役”的候选码。

相关概念：主属性和非主属性

  • 主属性：包含在任何一个候选码中的属性，都称为主属性。
  • 非主属性：不包含在任何候选码中的属性。

在我们的例子中：

  • 候选码是 (学号, 课程编号)。
  • 主属性是：学号 和 课程编号。
  • 非主属性是：姓名、系别、系主任、课程名称、成绩。

九、IDEF1X绘图规则

在 “确定型连接联系” 中，如果子女实体集中的每个实例都是由它与双亲的联系而确定，那么这个联系就被称为 “标定型联系”。在标定型联系中子女实体集总是从属实体集，标定型联系用一条实线或者线段把他们连接起来。父端实体集用圆角矩形框来表示，子女端用矩形来表示。

十、SQL

在 SQL 中，视图（view）是基于 SQL 语句的结果集的可视化的虚表，它包含行和列，就像一个真实的表。视图中的字段就是来自一个或多个数据库中的真实的表中的字段。我们可以向视图添加 SQL 函数、WHERE 以及 JOIN 语句。外模式也称子模式 (Subschema) 或用户模式，它是数据库用户（包括应用程序员和最终用户）能够看见和使用的局部数据的逻辑结构和特征描述，是数据库用户的数据视图，是与某一应用有关的数据逻辑表示。视图是外模式的实现方式之一。

十一、约束

UNIQUE、FOREIGN KEY 和 PRIMARY KEY 用于关系级别的约束；NOT NULL 用于列约束；CHECK 用于元组约束。

三级数据库技术1.数据库的三级模式结构和二层映像功能提供了数据的逻辑独立性和物理独立性，其中外模式/模式映像提供逻辑独立性，模式/内模式提供物理独立性。 内模式（最低层）：反映了数据在计算机物理结构中的实际存储形式。内模式也称存储模式或物理模式，它是数据物理结构和存储结构的描述，是数据在数据库内部的表示方式。并且一个数据库只有一个内模式。 模式（中层）：反映了设计者的数据全局逻辑要求。模式也称逻辑模式，是数据库中全体数据的逻辑结构和特征的描述，是所有用户的公共数据视图。它是数据库系统模式结构的中间层，不涉及数据的物理存储细节和硬件环境，与具体的应用和所使用的开发工具及高级程序设计语言无关。 外模式（最外层）：反映了用户对数据的要求。外模式也称子模式或用户模式，它是数据库用户（包括应用程序员和最终用户）看见和使用的局部数据的逻辑结构和特征的描述，是数据库用户的视图，是与某一应用有关的数据的逻辑表示。外模式是模式的子集，一个数据库可以有多个外模式。 2.概念数据模型：是数据库设计人员在设计的初始阶段摆脱计算机系统和DBMS的具体技术问题，集中精力分析数据以及...
